Resumen ejecutivo

El módulo ML de NormaBot clasifica descripciones textuales de sistemas de IA en cuatro niveles de riesgo definidos por el EU AI Act. Implementa dos backends complementarios con fallback automático y un override determinista basado en el Anexo III del Reglamento.

Manejo del desbalanceo de clases

# WeightedTrainer personalizado
class WeightedTrainer(Trainer):
    def compute_loss(self, model, inputs, return_outputs=False):
        labels = inputs.get("labels")
        outputs = model(**inputs)
        logits = outputs.get("logits")
        # Class weights calculados con scikit-learn
        class_weights = compute_class_weight(
            class_weight="balanced",
            classes=np.unique(labels.cpu()),
            y=labels.cpu()
        )
        loss_fct = CrossEntropyLoss(weight=torch.tensor(class_weights).float().to(logits.device))
        loss = loss_fct(logits, labels)
        return (loss, outputs) if return_outputs else loss

Data Augmentation

El dataset original de ~200 ejemplos es insuficiente para entrenar BERT desde cero. Se implementó un pipeline de aumentación de datos masivo que multiplica el dataset por un factor ~4–5x, preservando la semántica y la etiqueta de riesgo.

Dataset original

~200
ejemplos etiquetados manualmente

Dataset aumentado

~4000
original + paráfrasis + back-translation

Factor multiplicador

×5
1 original + 3 paráfrasis + 1 BT

Técnica 1: Paráfrasis con LLM (Bedrock Nova Lite)

El script paraphrase_llm.py usa Amazon Bedrock (Nova Lite v1) para generar n paráfrasis de cada ejemplo, con un prompt anclado a la etiqueta de riesgo para garantizar que la paráfrasis no cambie el nivel de riesgo de la descripción. 

# paraphrase_llm.py
model_id = "eu.amazon.nova-lite-v1:0"

prompt = f"""Genera {n} paráfrasis del siguiente texto sobre sistemas de IA.
Mantén el nivel de riesgo: '{risk_label}'.
No cambies el significado, sólo la expresión.
Texto: {text}"""

# Parámetros de generación
temperature = 0.8      # Suficiente variación léxica
max_tokens  = 1024

Técnica 2: Back-translation (ES → EN → ES)

El script back_translation.py traduce el texto al inglés y de vuelta al español usando GoogleTranslator (deep-translator), generando variaciones léxicas naturales sin alterar el significado. Los ejemplos donde la re-traducción es idéntica al original son descartados. 

# back_translation.py
from deep_translator import GoogleTranslator

def back_translate(text: str) -> str:
    en = GoogleTranslator(source="es", target="en").translate(text)
    es = GoogleTranslator(source="en", target="es").translate(en)
    # Filtrado: si es idéntico al original, descartar
    if es.strip() == text.strip():
        return None
    return es

Orquestador de aumentación

# run_augmentation.py
for example in original_dataset:
    output.append({**example, "source": "original"})

    # Paráfrasis (×3)
    for i, para in enumerate(generate_paraphrases(example["text"], n=3)):
        output.append({**example, "text": para, "source": f"paraphrase_{i+1}"})

    # Back-translation (×1)
    bt = back_translate(example["text"])
    if bt:
        output.append({**example, "text": bt, "source": "back_translation"})

# Guarda en dataset_augmented.jsonl (~4000 ejemplos)
Nota de calidad El prompt de paráfrasis incluye la etiqueta de riesgo como ancla para evitar que el LLM genere descripciones que cambien el nivel de riesgo (ej., de "inaceptable" a "alto riesgo"). Esto es crítico para preservar la integridad del dataset.

Rationale del override El modelo ML puede equivocarse en casos edge por insuficiencia del dataset. def predict_risk(text: str) -> dict:
    # 1. Validación (Pydantic) — min 1, max 5000 caracteres
    validated = _TextInput(text=text)

    # 2. Selección de backend + lazy loading thread-safe
    if CLASSIFIER_BACKEND == "bert":
        result = _predict_bert(text)       # fallback a XGBoost si falla
    else:
        result = _predict_xgboost(text)

    # 3. Override determinista Anexo III (post-procesado)
    result = _annex3_override(text, result)

    # 4. Observabilidad Langfuse
    langfuse.score(name="classifier_confidence", value=result["confidence"])

    return result

Limitaciones conocidas y decisiones de diseño

Dataset pequeño (limitación crítica documentada) El dataset original cuenta con ~200 ejemplos etiquetados manualmente. Esto requirió: data augmentation masivo (×5) para BERT, class_weight="balanced" en XGBoost, y cross-validation estricta (5-fold estratificado). El tamaño del dataset es una limitación reconocida en el proyecto.
#LimitaciónImpactoMitigación
L1 Dataset pequeño (200 ej. originales) Alto Data augmentation ×5, class_weight balanced, cross-val
L2 BERT F1 (0.729) inferior a XGBoost (0.882) Medio XGBoost como fallback y modelo principal. BERT backend semántico
L3 Sin explicabilidad SHAP para BERT Medio Confidence + probabilidades + backend XGBoost explicable
L4 Divergencia de patrones Art. 5 entre training e inferencia Medio Documentado en audit_errores [I1]. Override post-predicción actúa igual
L5 Sin reentrenamiento automático (CI/CD) Bajo retrain.py manual con política de promoción conservadora (≥0.005)
L6 Modelo BERT base anterior a 2024 (Art. 5.1.e post-AIA) Bajo Fine-tuning con datos manuales sobre Art. 5.1.e + override determinista

Decisiones de diseño destacadas

  • Lazy loading thread-safe: los modelos se cargan en el primer uso con double-check locking y threading.Lock, esencial para Streamlit multi-threaded.
  • F1-macro como métrica principal: captura el rendimiento en clases minoritarias, más representativo que accuracy para datasets desbalanceados.
  • Override post-predicción (no pre-procesado): el modelo ML siempre predice; el override solo corrige casos canónicos con certeza normativa absoluta.
  • LEAKAGE_COLUMNS explícito: frozenset centralizado en _constants.py que protege contra data leakage accidental durante el entrenamiento.
  • Dos backends paralelos: XGBoost más interpretable y preciso; BERT más semántico y robusto a variaciones léxicas. Fallback automático garantiza disponibilidad.
Auditoría de errores completada Se realizó una auditoría formal de errores (audit_errores_classifier.md). Los 3 errores críticos (alta severidad) fueron corregidos: centralización de _RISK_LABELS, corrección del índice en LogReg, y persistencia de pipeline_type en JSON.
